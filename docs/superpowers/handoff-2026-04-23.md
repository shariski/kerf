## Operational lessons learned in the prior session (save you ~30 min of trial and error)

1. **LSP diagnostics go stale after subagent file creation.** When a subagent creates a new module, the IDE's TypeScript indexer may report "Cannot find module './newModule'" for ~30 seconds. Trust `npm run typecheck` exit code over IDE diagnostics. Real state is the source of truth.

2. **rtk proxy intercepts `npx`.** `npx tsx scripts/foo.ts` fails with "Missing script: 'tsx'" because rtk routes it as `npm tsx`. Use `./node_modules/.bin/tsx` for direct binary invocation. `npx vitest` similarly fails — use `npm test -- <path>` instead.

3. **Model selection cheat sheet:**
   - **Haiku**: pure-domain TDD with full plan code blocks (Tasks 1, 2, 4, 5, 6, 7, 8 worked great with haiku). DB migrations. Doc-only fixes. Tiny code-review-driven fixes.
   - **Sonnet**: UI work in large existing files (Task 3 onboarding.tsx was 24K — sonnet handled it). Multi-file integration where pattern-matching matters. Anything where the plan provides a skeleton but not literal code.
   - **Opus**: not needed for any task in this plan if the spec is faithful.

4. **Two-stage review is genuinely worth it.** Five real defects were caught and fixed in-branch before PR:
   - Task 2: `computeBaseline` cold-start path discarded the journey parameter
   - Task 3: typecheck errors (TS2532) on `mock.calls[0][0]` access; aria-label on radios overrode implicit label and hid descriptions from screen readers
   - Task 5: docstring overclaimed score comparability between motion-pattern and character/bigram scores
   - Task 6: redundant `.length === 0` checks; lenient `.toContain` test that should have been `.toBe`
   - Task 8: `SessionTarget.score` returned unweighted candidate score instead of journey-weighted product (would have broken Task 10 + the `session_targets.selection_score` DB column)
   
   Spec review catches "did the code implement what's named in the spec." Code-quality review catches "does the code's behavior match what callers will expect, beyond the spec's literal text." Both are needed.

5. **Pre-existing Playwright failure** (`tests/a11y.spec.ts`) is unrelated to ADR-003 and present on main. Don't get distracted by it. Document it in test plans as "known failure."

6. **Per-task subagent invocation pattern that worked**:
   - Implementer prompt = full task text + scene-setting context + behavior format (Status: DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT) + commit message
   - Spec reviewer prompt = original spec + implementer's claim + verification commands + "DO NOT trust the report" instruction
   - Code reviewer prompt = use `superpowers:code-reviewer` subagent_type with template at `requesting-code-review/code-reviewer.md` — focus on architecture/quality, NOT spec compliance (already verified)
   - Fix-loop subagent = narrow, surgical, only the changed file(s), explicit commit message

7. **Branch + PR cadence:** small slices ship faster than big slices. Each PR has 2–5 tasks based on natural seams. PR titles: `feat(adr-003): <slice name> (Tasks X–Y)`. PR body uses HEREDOC with Summary + Test plan + References.

8. **Conventional commit format** (preserved across all 14+ commits in PRs #48 and #49):
   ```
   feat(adaptive): <one-line summary> (ADR-003 §X)
   
   <body explaining the why, not the what>
   
   Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
   ```
   
   Use `feat(scope):`, `fix(scope):`, `refactor(scope):`, `chore(scope):`, `docs(scope):` per the conventional-commit style. Commit messages are passed via HEREDOC in `git commit -m "$(cat <<'EOF' ... EOF)"`.

## Subagent dispatch patterns (calibrated leaner than skill defaults)

The skill `superpowers:subagent-driven-development` has defaults that are deliberately verbose for defect-prevention. Across 8 tasks, several were tunable downward without losing review value. Use these tuned patterns by default in this codebase; revert to skill-purist if a pattern misfires.

### Pattern 1: Reference plan by path + line range (don't paste verbatim)

The skill's red flag "never make subagent read plan file" assumes the alternative is the subagent navigating the file system. Here the plan has stable section markers and you can hand it the exact line range. Subagent does ONE Read with offset/limit — same reliability as paste, no double-flow of the spec text through context.

**Use:**
```
Read these exact lines — full Task N spec:
File: docs/superpowers/plans/2026-04-22-deliberate-practice-architecture.md
Lines: <start>–<end>

Follow every step in that range verbatim.
```

**Saves**: ~300–500 lines per implementer prompt; same per reviewer prompt.
**Risk**: subagent skims past part of the spec. Mitigation: spec reviewer verifies actual code matches plan content via `git show <commit> -- <file>`.

### Pattern 2: Skip spec review for pure-data / pure-copy-paste tasks

Spec review caught zero issues across pure-data tasks (Task 7 specifically). Code-quality review found nothing either, but at least flagged the *possibility* of issues (e.g. content typos, factory abstractions). For these tasks, the spec reviewer is pure overhead.

**Skip spec review when the task is**:
- Pure data authoring with structural validation built into the implementation tests
- Verbatim copy-paste from plan with zero judgment calls

**Keep spec review when the task involves**:
- Multi-file changes
- Branching logic
- UI / component work
- Anything where "what's requested" might be misread

**Saves**: ~1 reviewer subagent per applicable task.

### Pattern 3: Inline trivial fixes (skip the fix-subagent for ≤3-line fixes)

The skill's "don't fix manually" rule is good for non-trivial fixes (where context pollution risks bias). For 1–3 line edits with no semantic ambiguity (a `!` non-null assertion, a redundant-check removal, a one-line doc tweak), dispatch overhead dwarfs the fix.

**Apply fixes inline when ALL hold**:
- ≤3 lines changed in 1 file
- No semantic ambiguity (just typing/syntax)
- Reviewer's fix instruction is fully explicit
- You write the commit message yourself

**Otherwise dispatch a fix subagent.**

After inline fix: run typecheck + the relevant test file before committing. Next-slice PR review still catches anything semantic that slipped.

**Saves**: ~1 fix subagent per applicable case (typically 1–2 cases per slice).

### Pattern 4: Compact return format from subagents

Implementer + reviewer reports were ~100–200 lines each, mostly restating context. Tell each subagent to constrain return length explicitly.

**In every implementer prompt, include**:
> Report format (max 25 lines):
> Status (DONE/DONE_WITH_CONCERNS/BLOCKED/NEEDS_CONTEXT) | commit SHA | test count pass/fail | typecheck status | 1-line concerns if any | 1-line list of changed files

**In every reviewer prompt, include**:
> Report format (max 30 lines):
> Verdict (✅/❌) | issues by severity (file:line + 1-line description per issue) | re-test required y/n | 1-line list of files inspected

**Saves**: ~75 lines per subagent return × ~6 subagents per slice = ~450 lines.

### Pattern 5: Trust LSP-staleness pattern; don't reflexively re-verify

The IDE TypeScript indexer goes stale after subagent file creation. The pattern is consistent: "Cannot find module './X'" + cascade ("Parameter implicitly has 'any' type" on params from the unfound module) ≡ stale.

**Re-verify with `npm run typecheck` only when**:
- Implementer didn't claim typecheck passed, OR
- IDE diagnostic is something OTHER than "Cannot find module" / its cascade

**Don't re-verify when**:
- Implementer claims typecheck 0 errors AND
- IDE diagnostic is the recognizable stale pattern

**Saves**: ~3–5 bash calls per slice.

### What to NOT lean down

| Pattern | Why keep skill-purist |
|---|---|
| Implementer + code-quality reviewer for logic tasks | Caught 5 real defects in 8 tasks. High signal. |
| Worktree-per-slice | Skill requirement; isolation matters |
| Conventional commit + HEREDOC + Co-Authored-By trailer | Load-bearing for git archaeology |
| Per-slice PR with review-loop fixes in-branch | Clean history; one merge unit |
| `superpowers:code-reviewer` subagent type for code-quality review | Standardized review template |

### Expected savings per slice

Roughly **2,000 lines of token traffic per slice** with all five patterns active. Wall time reduction is smaller — maybe 5–10 min per slice — because most time is subagent thinking, not prompt size.

### When to revert a pattern

If a pattern misfires (e.g. subagent skips part of plan because it didn't read carefully under Pattern 1), revert THAT pattern only and report what failed. Don't roll back all patterns from one miss.
