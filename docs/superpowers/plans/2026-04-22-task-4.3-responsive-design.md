# Task 4.3 — Responsive Design — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Gate kerf to viewports `≥ 768px`. Below that, render a bare full-screen "kerf is a desktop experience" message; above that, render the existing desktop UI unchanged.

**Architecture:** Pure CSS-only solution. Mount a `<MobileGate>` component as a sibling of the app tree in `__root.tsx`; a `@media (max-width: 767px)` query decides which one displays. No JS viewport detection, no SSR hydration swap, no per-route guards. Every route is gated uniformly — including `/login` and `/onboarding`.

**Tech Stack:** React + Tanstack Start (`__root.tsx`), hand-rolled `kerf-*` CSS in `src/styles.css`, Vitest for unit tests, Playwright for viewport visual checks.

```tsx
/**
 * Mobile gate — Task 4.3.
 *
 * Rendered as a sibling of the app tree in __root.tsx; a CSS media
 * query at max-width: 767px hides the app and shows this gate. No
 * props, no state, no interactivity — kerf is a split-keyboard
 * practice platform and a touch screen literally cannot host the
 * product. Firm and respectful per CLAUDE.md §B3 accuracy-first tone.
 */

export function MobileGate() {
  return (
    <main
      className="kerf-mobile-gate"
      aria-labelledby="kerf-mobile-gate-headline"
    >
      <div className="kerf-mobile-gate-inner">
        <div className="kerf-mobile-gate-logo" aria-hidden>
          kerf.
        </div>
        <h1
          id="kerf-mobile-gate-headline"
          className="kerf-mobile-gate-headline"
        >
          kerf is a desktop experience.
        </h1>
        <p className="kerf-mobile-gate-body">
          You'll need a split mechanical keyboard to practice — we'll see you
          at your desk.
        </p>
      </div>
    </main>
  );
}
```

```
## Summary
- Below 768px, kerf renders a bare full-screen gate: "kerf is a desktop experience."
- At or above 768px (tablet + desktop), the existing UI renders unchanged
- Pure CSS-only implementation — no JS viewport detection, no SSR hydration swap

## Why this shape
kerf is a practice program for split mechanical keyboards. Mobile devices literally cannot host the product, so the gate is genuine product-gating rather than a "sorry, not yet" placeholder. Tablet widths (768–1024px) are above the threshold because an iPad paired with an external split keyboard is a real setup.

## Scope
Per docs/03-task-breakdown.md §4.3: Desktop primary / Tablet acceptable / Mobile: "use desktop" message.

## Copy (locked per CLAUDE.md §B3 accuracy-first tone)
- Headline: `kerf is a desktop experience.`
- Body: `You'll need a split mechanical keyboard to practice — we'll see you at your desk.`

## Out of scope (explicit non-goals)
- Tablet-specific layout tweaks (stacked cards, etc.)
- Marketing content or landing blurbs on the gate
- "Continue anyway" escape hatch
- UA sniffing / server-side viewport detection
- /login or any other route-level exemption — the gate applies uniformly

## Known edge case
Recharts charts on the dashboard may render with zero width if a user loads at mobile, then resizes into desktop range without a reload. Reload fixes it. Not worth a ResizeObserver workaround in Phase A.
```

## Risks to watch during execution

- **The required CSS tokens are confirmed present** (Task 3, Step 1 guards against any drift since the spec was written).
- **`<Scripts />` placement** — keep it inside `.kerf-app-root`; moving it out of the app subtree would either hydrate the gate (needless) or break desktop hydration.
- **`<main>` tag duplication** — `__root.tsx` renders `<AppNav />` + `{children}` inside the app-root div; some routes (`/dashboard`, `/practice`) use `<main>` as their top-level element. The gate also uses `<main>`. This is only a problem if both are visible at once, which the CSS media query prevents. HTML validation may still warn about two `<main>` elements in the source DOM at any viewport; that's acceptable. If the validator warning proves unacceptable, switch the gate to `<div role="main">` — functionally identical for screen readers. Defer unless flagged in review.
