# Task 4.5 — Documentation pages — Design

**Status:** draft → pending user review
**Target PR branch:** `feat/task-4.5-docs` (off `main`)
**Scope:** `docs/03-task-breakdown.md §4.5` — "How it works", "Why split keyboard transition is hard", FAQ, Privacy + ToS templates. Per IA `05 §3` these are footer-link pages, not primary nav. Task also introduces the project's first global footer since none exists today.

## Why

kerf is niche. The users who land here need to know *whether this
product is for them* in under a minute, and *how the engine makes
decisions* because the accuracy-first + transition-phase positioning
is unusual. Without docs, every pre-sign-up question ("is this just
Monkeytype?", "why does it say I'm getting slower?", "does it work
with my Corne?") has to be answered ad hoc. The docs also anchor the
legal requirements before beta launch (Task 4.7).

## User decisions locked in (2026-04-22)

- **Route structure:** 5 separate routes, not tabs or anchors —
  `/how-it-works`, `/why-split-is-hard`, `/faq`, `/privacy`, `/terms`.
  Separate URLs so each page is deep-linkable and independently
  shareable. Legal docs especially benefit from stable URLs.
- **Content depth:** medium (~800–1,200 words per explainer,
  ~10-12 questions in FAQ). Enough to earn trust with a niche audience
  that expects transparency (§2.4 + §B3); short enough that nobody
  thinks it's homework.
- **Legal templates:** generic templates with `{{DOUBLE_BRACE}}`
  placeholders the developer fills in before deploy. Claude Code
  does not make up business-specific legal content.
- **Footer:** new global `<AppFooter>` component. Rendered by
  `__root.tsx`, suppressed on the same chromeless routes as `AppNav`
  (`/onboarding`, `/login`) + on active-typing states where focus is
  paramount.

## Non-goals (explicitly out of scope)

- Blog / changelog / roadmap pages.
- Dynamic content loading (no CMS, no Markdown renderer).
- Internationalization — English-only for Phase A.
- "What's new" or release-notes page.
- Embedded video / illustrations / images.
- Converting internal product docs (`docs/01-*.md` through
  `docs/06-*.md`) to public pages. Those are for the development team.
- Stat counters or animated hero sections on the doc pages.
- Newsletter / email capture.

## Design

### Routes

Five server-rendered static routes. Tanstack Start file-based routing:

- `src/routes/how-it-works.tsx`
- `src/routes/why-split-is-hard.tsx`
- `src/routes/faq.tsx`
- `src/routes/privacy.tsx`
- `src/routes/terms.tsx`

Each is a default-exported route component wrapped in a shared
`<DocPage>` layout. No loaders, no server fns, no client state.

### Shared `<DocPage>` component

Location: `src/components/doc/DocPage.tsx`.

Props:
```ts
type Props = {
  title: string;
  /** Optional short subtitle / lede under the h1 */
  lede?: string;
  /** Effective date shown above legal-template pages */
  effectiveDate?: string;
  /** Shown as a yellow bar at the top of template pages */
  isTemplate?: boolean;
  children: React.ReactNode;
};
```

Renders a 720px-max reading column (matches
`.kerf-practice-container` precedent), the `<h1>` with the title, an
optional lede, the children, and — if `isTemplate` — a distinctive
amber-bar notice at the top that reads `This page is a template.
Replace the highlighted placeholders before deploying to production.`

Uses `<main id="main-content">` per the Task 4.4 contract.

### Content style rules (per §B3)

- No exclamation marks.
- Never "Welcome to kerf!" or equivalent hype. Open with a fact.
- The word "amazing" and its cousins are banned.
- When explaining the engine, we lead with *what* the user sees, then
  explain *why* the engine picks that.
- Stats framing: accuracy first, speed second.
- First-person plural ("we") only for product decisions; everything
  else is written in second person ("you").

### Content outline per page

#### `/how-it-works`

Sections:
1. **The short version.** 2-3 sentence summary: word-picker +
   weakness score + transition phase + split metrics.
2. **What the engine notices.** Keystroke events → character &
   bigram stats → decayed 30-day weight.
3. **Picking the next exercise.** The weakness score formula
   (phase-aware coefficients), the inner-column transition bonus,
   how selection becomes a word pick from the corpus.
4. **Why it runs two ways.** Transitioning vs refining — different
   coefficient profiles + different generation priorities. Honest
   about when the engine decides to switch you.
5. **What it deliberately doesn't do.** No LLM generation, no
   server-side content pick, no hidden "gamification" score.
6. **A worked example.** Dev-friendly: "here's a mid-session
   transitioning user with weak B, N, T; the engine ranks these
   with scores 3.8 / 3.3 / 2.6 and picks words with those letters
   plus inner-column density."

#### `/why-split-is-hard`

Sections:
1. **Why split keyboards break muscle memory.** Columnar vs
   staggered; new inner-column hand positions; thumbs doing work
   they've never done.
2. **Why typing-speed apps don't help.** Monkeytype-style training
   rewards speed early; for a transitioner, chasing speed first
   locks in errors. We're counter-programming against that.
3. **What actually changes physically.** Index-finger reach
   inversions (e.g. B on Sofle lives in a different place than on
   QWERTY). Thumb cluster. Cross-hand bigrams shift timing.
4. **What kerf does differently.** Weakness scoring that values
   accuracy over speed, phase-aware coefficients, split-specific
   metrics (inner-column error rate, thumb cluster time).
5. **Who this is not for.** Copy of the "NOT for" list from product
   spec §3 — honest exclusions.

#### `/faq`

~10-12 questions:

- Do I need a specific keyboard? — Sofle and Lily58 are supported in
  Phase A.
- Does this work with Corne / Ergodox / Moonlander / Glove80? — Not
  yet; see the roadmap answer.
- Does it work with Colemak or Dvorak? — Phase A is QWERTY-only.
- Why not just use Monkeytype? — Two reasons: split-specific metrics;
  accuracy-first pacing.
- How is my data used? — Stored per-account; not sold, not shared;
  see Privacy for details.
- Why does the app say I got slower? — Speed drops during transition
  are expected; that's the point.
- What is "transitioning" vs "refining"? — The two engine phases.
- Can I turn off the adaptive engine and just drill? — Yes — drill
  mode.
- How long until I'm fast on my split? — Honest answer: depends.
  Typical range from beta: 2–8 weeks.
- Is my typing data ever shared? — No.
- Can I export my data? — Not yet; V2.
- Is there a mobile app? — No — see the desktop gate.

Exact wording settles during implementation; these are the topics.

#### `/privacy`

Standard structure with `{{PLACEHOLDER}}` markers:

- Effective date ({{EFFECTIVE_DATE}})
- Who we are ({{COMPANY_NAME}}, {{JURISDICTION}}, {{CONTACT_EMAIL}})
- What we collect (email via better-auth magic links; session
  keystrokes + stats per account; no third-party analytics, no cookies
  beyond session auth)
- How we use it (adaptive engine, dashboard stats, nothing else)
- Storage (in region, {{DATA_REGION}})
- Your rights (access, deletion request to {{CONTACT_EMAIL}})
- Third parties used ({{EMAIL_PROVIDER}} — Resend; {{HOSTING}}; nothing
  else)
- Changes to this policy
- Contact

#### `/terms`

Standard structure with placeholders:

- Effective date
- Who we are
- Service description (what kerf is + what it isn't)
- Account (how magic-link login works; user responsibilities)
- Acceptable use (no scraping, no automated abuse; honest typing)
- Intellectual property (kerf owns the site, user owns their typing
  data)
- Disclaimers + limitation of liability ({{JURISDICTION}}-appropriate)
- Termination
- Governing law ({{JURISDICTION}})
- Contact ({{CONTACT_EMAIL}})

### Global footer

Location: `src/components/nav/AppFooter.tsx`.

Renders two rows of text links + a copyright line:

```
  how it works   ·   why split is hard   ·   faq
  privacy   ·   terms
  © 2026 kerf
```

- `--font-mono` 12px, `text-secondary` color, amber hover.
- Hidden on chromeless paths (`/onboarding`, `/login`) — same list as
  `AppNav` in `__root.tsx`. Task's implementation passes the
  chromeless decision down once.
- Hidden on active-typing states. Practice & drill routes render
  different `<main>` containers per stage (`kerf-practice-main` vs
  `kerf-practice-main--active`). We'll conditionally render
  `<AppFooter />` only when the current route is not in an active
  typing stage. Simpler: add a `data-stage` attribute to
  `.kerf-app-root` and let CSS hide the footer when `data-stage="active"`.

Footer lives inside `.kerf-app-root` so the CSS-only mobile gate
(Task 4.3) already hides it correctly on sub-768px viewports.

## Risks

- **Footer stage-based hiding is state-ish.** Simpler alternative
  if it gets complicated: just always show on `/practice` and rely
  on the typing-area layout to dominate visually. Punt to the
  implementation plan — pick the simplest thing that works.
- **Copy tone drift.** Writing ~4,000 words of user-facing copy is
  the single largest surface for accidental hype/excitement to leak
  in. Self-check pass at the end: grep for banned words (`amazing`,
  `incredible`, `crushing`, `!`) and re-read.
- **Legal template maintenance.** Placeholders are clearly marked,
  but a developer who forgets to replace them before deploy is a
  real risk. Mitigated by (1) visible amber banner on the page,
  (2) inline comment at top, (3) `docs/legal-templates.md`
  checklist, (4) grep command in the pre-deploy checklist added to
  README.
- **~4,000 words is enough that the bundle grows.** Static JSX is
  cheap; but if the size noticeably bloats the 1-byte-matters part
  of the bundle we can split out content into a lazy-loaded route.
  Unlikely to matter in Phase A.
