# Task 4.3 — Responsive design — Design

**Status:** draft → pending user review
**Target PR branch:** `feat/task-4.3-responsive` (off `main`)
**Scope:** `docs/03-task-breakdown.md §4.3` — Desktop primary / Tablet acceptable / Mobile: "use desktop" message.

## Why

kerf is a practice program for split mechanical keyboards. Mobile screens
literally cannot host the product: no physical split keyboard, no useful
keyboard-first shortcuts, no typing-session fidelity. A small percentage of
visits will come from mobile (shared links, curiosity, marketing); we owe
them a clear message rather than a broken experience. Tablets (iPad+, Surface)
are a plausible second-surface use case when paired with an external split
keyboard, so the desktop UI is allowed to scale down to tablet widths rather
than gate there.

## User decisions locked in (2026-04-22)

- **Breakpoint strategy:** two-bucket — `< 768px → mobile gate`; `≥ 768px → desktop UI (same layout, scales narrower on tablet)`.
- **Gate style:** bare — full-screen message only. No navigation, no marketing copy, no escape hatches.
- **Exemptions:** none — the gate applies to every route uniformly (including `/login`, `/onboarding`).
- **Mechanism:** CSS-only `@media (max-width: 767px)`. No client-side JS viewport detection. No SSR hydration flicker.

## Non-goals (explicitly out of scope)

- Tablet-specific layout adjustments (stacked cards, collapsed sidebars). `§B6 Phase A scope discipline` — no speculative tablet work; if real tablet feedback surfaces pain points post-beta, scope a follow-up.
- Marketing content, "what is kerf" blurbs, or email capture on the gate.
- "Continue anyway" escape hatch or `?force=desktop` query overrides.
- Any change to wireframe-level desktop layout or breakpoint-aware reflow of existing components.
- UA sniffing or server-side viewport detection.
- Dashboard chart re-measurement on viewport change (known edge case — see Risks).

## Design

### Component: `<MobileGate>`

Location: `src/components/MobileGate.tsx`

Renders a full-viewport message. Static content, no props, no state.

```tsx
<main className="kerf-mobile-gate" aria-labelledby="kerf-mobile-gate-headline">
  <div className="kerf-mobile-gate-inner">
    <div className="kerf-mobile-gate-logo">kerf.</div>
    <h1 id="kerf-mobile-gate-headline" className="kerf-mobile-gate-headline">
      kerf is a desktop experience.
    </h1>
    <p className="kerf-mobile-gate-body">
      You'll need a split mechanical keyboard to practice — we'll see you at
      your desk.
    </p>
  </div>
</main>
```

**Copy (locked, per `CLAUDE.md §B3`):**
- Headline: `kerf is a desktop experience.`
- Body: `You'll need a split mechanical keyboard to practice — we'll see you at your desk.`

No exclamation marks, no "Sorry", no apology. Firm + respectful + short.

### Mount point

In `src/routes/__root.tsx`'s `RootDocument`, add the `MobileGate` as a sibling
of the existing body content; wrap the existing content in a `.kerf-app-root`
container so CSS can toggle the pair:

```tsx
<body>
  <MobileGate />
  <div className="kerf-app-root">
    {!chromeless && <AppNav />}
    {children}
    {import.meta.env.DEV && <DevtoolsLazy />}
    <Scripts />
  </div>
</body>
```

`<Scripts />` must remain inside `.kerf-app-root` so hydration scripts don't
load twice if the browser later resizes the viewport up into desktop range.

### Accessibility

- `<main role="main">` via the implicit semantic on `<main>` — screen readers announce the gate as the page content.
- `aria-labelledby` ties the main region to the headline.
- All four CSS variable references (`--color-kerf-bg-base`, `--color-kerf-text-primary`, `--color-kerf-text-secondary`, `--color-kerf-amber-base`) are existing design-system tokens with WCAG AA contrast against the espresso base.
- No animations — `prefers-reduced-motion` is a no-op but kept globally consistent.

### SEO / crawling

- Googlebot renders at desktop widths by default and will see the app.
- Mobile crawlers will see the gate, which still has meaningful text content (headline + body) — not an empty page.

- 768×1024 (iPad portrait) — app visible, gate hidden, existing desktop UI renders
- 1280×800 (laptop) — app visible, gate hidden, no regression vs. current state

**Unit test** (`src/components/MobileGate.test.tsx`):
- Renders headline text exactly `"kerf is a desktop experience."`
- Renders body text exactly `"You'll need a split mechanical keyboard to practice — we'll see you at your desk."`
- No links, no buttons (interactive-element count is zero)

**Manual check:**
- Resize a desktop browser window across the 768px boundary; the swap is instant, no flicker, no layout shift.
- Visit `/login`, `/practice`, `/dashboard`, `/keyboards`, `/`, `/onboarding`, `/practice/drill` on a narrow viewport — all show the same gate.

## Risks

- **Recharts re-measurement on viewport change:** If a user loads the site at mobile width, then rotates/resizes into desktop range, the dashboard's Recharts charts (TrajectoryCharts, TemporalPatterns) may measure zero-width containers from the brief mobile render and render empty. Mitigation: reload fixes it; Phase A acceptable; document in PR description as known edge case. Not worth a ResizeObserver workaround in MVP.
- **Prefer `100svh` over `100vh`:** mobile browsers shrink `100vh` when the URL bar is visible. Using `100svh` (small viewport height) avoids the "gate text gets pushed below fold" jumpiness. Fallback for older browsers is acceptable — the gate is still readable, just slightly cropped.
- **CSS token coverage:** all tokens used (`--color-kerf-bg-base`, `--color-kerf-text-primary`, `--color-kerf-text-secondary`, `--color-kerf-amber-base`, `--font-brand`) are confirmed existing in `styles.css`. No design-system additions required.
- **iPad landscape (1194×834):** above the 768px threshold — treated as desktop. Sane default: iPad in landscape with external keyboard is a genuine use case.
